# SpringBoot工程启动以后，我希望将数据库中已有的固定内容，打入到Redis缓存中，请问如何处理？

现在主流的方案有三种：

  1. CommandLineRunner / ApplicationRunner

    实现这俩接口中的任意一个，Spring Boot 会在所有 Bean 初始化完成、应用正式启动之后回调 run 方法。这时候去查库写缓存，时机刚刚好。两者的区别就是参数不同，CommandLineRunner 拿到的是原始 String 数组，ApplicationRunner 拿到的是封装过的 ApplicationArguments，功能上没本质差别。

    @Component
public class CacheWarmUpRunner implements CommandLineRunner {
    @Autowired
    private ProductRepository productRepository;
    @Autowired
    private StringRedisTemplate redisTemplate;

    @Override
    public void run(String... args) {
        // 查出所有热门商品，一次性灌入缓存
        List<Product> hotProducts = productRepository.findHotProducts();
        for (Product p : hotProducts) {
            redisTemplate.opsForValue().set("product:" + p.getId(), JSON.toJSONString(p), 30, TimeUnit.MINUTES);
        }
    }
}

# SpringBoot配置文件加载优先级你知道吗？

Spring Boot 采用 PropertySource 机制来管理配置，后加载的配置源会覆盖先加载的。

主要关注的配置优先级从高到低排列如下：

命令行参数 > JAR包外的 application-{profile}.properties > JAR包外的 application.properties > JAR包内的 application-{profile}.properties > JAR包内的 application.properties

对于 application 配置文件本身，还有一个位置优先级。Spring Boot 会按以下顺序搜索配置文件，后面的位置覆盖前面的：

  1. classpath 根目录
  2. classpath 下的 /config 目录
  3. 当前运行目录
  4. 当前运行目录下的 /config 子目录
  5. /config 子目录的直接子目录

当 application.properties 和 application.yml 同时存在于同一位置时，最终生效的是 application.properties 中的配置

# SpringBoot是否可以使用XML配置？

可以用，但不推荐。Spring Boot 的设计理念是约定大于配置主推 Java 注解和 properties.yml 文件的方式。

如果确实需要引入 XML 配置，用 @ImportResource 注解就能把 XML 配置文件加载进来。

# SpringBoot中如何实现定时任务？

Spring Boot 里实现定时任务最简单的方式就是 @Scheduled 注解

  1. 在启动类上加 @EnableScheduling 开启定时任务功能

    @SpringBootApplication
@EnableScheduling
public class MyApplication {
    public static void main(String[] args) {
        SpringApplication.run(MyApplication.class, args);
    }
}

@Scheduled 有三种常用的执行策略：

  1. fixedRate

    是固定速率执行，理论上时间一到就开始下一次。但 Spring 默认是单线程调度器，如果上一个任务还没执行完，下一个任务必须等上一个任务执行完才能开始。

  2. fixedDelay

    是固定延迟执行，上一次执行完才开始计时。

  3. cron

    表达式最灵活，可以指定具体的执行时间点。

# @Async什么时候会失效？

Spring 的 @Async 底层是通过 AOP 代理来实现的，只有调用主代理对象的方法，才能被拦截然后丢到线程池里执行。任何绕过代理的情况，都会让 @Async 变成摆设。

最常见的失效场景有这么几种：

  1. 同一个类里内部调用

    在 Service 的 A 方法里直接调 this.B()，这个 B 方法上的 @Async 压根不生效。因为 this 指向的是原始对象，不是代理对象，调用根本没经过 AOP 拦截。

    @Service
public class OrderService {
    public void createOrder() {
        // 失效！this 指向原始对象，不是代理
        this.sendNotification();
    }

    @Async
    public void sendNotification() {
        // 期望异步执行，但实际是同步的
    }
}

  2. 方法不是 public 的

    Spring 默认用的是基于接口的 JDK 动态代理或者基于类的 CGLIB 代理，这两种代理都只能拦截 public 方法。private、protected、包级别的方法，代理根本拦不到。

  3. 压根没开启异步支持。@Async 不是开箱即用的，必须在配置类或启动类上加 @EnableAsync，Spring 才会去扫描 @Async 注解并创建代理。不加这个注解，@Async 就是个装饰品。

  4. Bean 不是 Spring 管理的。自己 new 出来的对象，Spring 根本不知道它的存在，更别提给它创建代理了。

  5. 返回值类型不对。@Async 方法的返回值只能是 void、Future、CompletableFuture 这几种。你要是返回个普通 String，虽然不会报错，方法也执行了，但调用方拿到的是 null，异步执行的结果你拿不到，看起来好像失效了。

# 说说Springboot的启动流程？

关于 Spring Boot 的启动流程，其实核心就做两件事：构建对象和运行启动。

我们可以把它想象成“准备开一家餐厅”的过程。以 2.7.6 版本为例，主要流程如下：

  1. 第一阶段：准备工作（new SpringApplication）

    就像开店前要先确定我们是开火锅店还是奶茶店。

    1. 确定应用类型：Spring Boot 会自动判断当前是 Web 应用（Servlet）、响应式应用（Reactive）还是普通应用。
    2. 加载初始化器和监听器：利用 SPI 机制（读取 spring.factories），把所有需要用到的初始化器和监听器都找出来存好，随时待命。

  2. 第二阶段：正式启动（run 方法）

    这是最核心的阶段，就像点火开业，主要分五步走：

    1. 准备环境（Environment）：先把配置文件 application.yml（菜单）、环境变量（天气）、命令行参数（顾客要求）都加载好，组成一个环境对象。
    2. 创建容器（ApplicationContext）：根据第一步确定的类型，创建一个空的 Spring IOC 容器（比如 AnnotationConfigServletWebServerApplicationContext）。
    3. 刷新容器（Refresh Context），这是最重要的一步。Spring 会解析 @SpringBootApplication，扫描所有的 Bean，执行自动配置（AutoConfiguration），把所有组件注入到容器里。关键点：如果是 Web 应用，内嵌的 Tomcat 就是在这个环节（onRefresh 方法）启动的。
    4. 收尾工作：容器刷新完成后，发布启动完成事件，通知所有监听器。
    5. 执行回调（Runners）：最后，检查有没有实现了 CommandLineRunner 或 ApplicationRunner 的 Bean，如果有，就执行它们的 run 方法。这通常是我们写开机自启任务的地方。

# SpringBoot是如何实现自动配置的？

Spring Boot 自动配置的核心就是 @EnableAutoConfiguration 注解，它通过 @Import 引入了 AutoConfigurationImportSelector，这个类会去扫描 classpath 下所有 jar 包里的 META-INF/spring.factories 文件，找出里面声明的自动配置类，然后根据条件决定哪些配置要加载。

整个流程是这样的：Spring Boot 启动时，@SpringBootApplication 里包含的 @EnableAutoConfiguration 生效，触发 AutoConfigurationImportSelector 去读取 spring.factories，拿到一堆候选配置类，然后逐个检查这些类上的 @ConditionalOnClass、@ConditionalOnMissingBean、@ConditionalOnProperty 这些条件注解，只有条件全部满足的配置类才会真正加载，往容器里注册 Bean。

# SpringBoot（Spring）中为什么不推荐使用@Autowired？

准确说不是不推荐用 @Autowired，而是不推荐用 @Autowired 字段注入，Spring 官方文档明确推荐的是构造器注入。

因为字段注入可能会触发空指针问题，且对单元测试不友好，并且会掩盖循环依赖问题。

还有 @Autowired 实际上 Spring 提供的，导致了业务代码和框架强绑定，如果更换给其他 IOC 框架则不适用了，而 @Resource 是 JSR-250 提供的，它是 Java 的标准，因此如果非要使用字段注入也应该使用 @Resource。

字段注入的问题在于：依赖是在对象创建之后才被“偷偷”塞进去的，所以会带来一些问题（详细看扩展知识）。

# 如何在SpringBoot启动时执行特定代码？有哪些方式？

SpringBoot启动时执行代码，本质上是利用它的生命周期回调机制。你不需要手动触发，框架会在启动的不同阶段自动调用你注册的逻辑。

最常见的办法是实现 CommandLineRunner 接口。只要你的类加上 @Component，重写 run 方法，它就会在容器初始化完成后执行。参数就是应用启动时传入的命令行参数。

@Component
public class MyStartupTask implements CommandLineRunner {
    public void run(String... args) {
        System.out.println("启动任务执行");
    }
}

# SpringBoot中如何实现异步处理？

Spring Boot 的异步处理主要靠 @Async 注解和线程池配合，把耗时操作从主线程剥离，避免阻塞请求响应。

  1. 先在启动类或配置类上加 @EnableAsync 开启异步支持。然后在需要异步执行的方法上标注 @Async，这个方法就会被动态代理拦截，交给独立线程去跑。
  2. 默认用的是 Spring 内置的 SimpleAsyncTaskExecutor，但生产环境一定得自定义线程池。比如用 ThreadPoolTaskExecutor 配置核心线程数、队列容量这些参数，不然容易把系统搞崩。常见的像发短信、写日志这种非关键路径操作，就适合扔进异步处理。
  3. 返回值可以是 void 或 Future / CompletableFuture。如果要获取结果，推荐用 CompletableFuture，它支持链式调用和组合，跟业务逻辑融合更顺。

举个例子，用户注册后异步发送欢迎邮件，主线程完成注册直接返回，邮件任务由专门的线程池处理，这样接口响应能从几百毫秒降到几十毫秒。

HTTP请求 → 主线程处理核心逻辑 → @Async标记方法 → 任务提交到线程池 → 异步线程执行耗时任务 → 快速返回响应

注意事务和异常的处理。@Async 方法别直接抛异常，最好内部 try-catch 记录日志，因为调用方可能已经走完了。另外，同一个类里方法直接调用不会触发代理，得拆到不同 Bean 里。

# 在SpringBoot中如何实现多数据源配置？

推荐使用mybatis-plus的多数据源插件，支持读写分离，分库分表的场景

实在想要自己实现可以按照以下思路去做： 1.配置多个数据源的配置文件 2.排除springboot自动加载的数据源 3.为每个数据源都配置好自己的 DataSource 和 事务管理器 4.使用 AbstractRoutingDataSource 通过标识key进行动态路由切换,可以手动，也可以通过aop根据注解动态切换

配置多数据源主要解决一个应用需要访问多个数据库的场景，比如订单库和用户库分离。关键在于让 Spring 知道哪个方法该用哪个数据源。

Spring Boot 默认只配一个数据源，多个就得手动干预。核心是动态数据源路由，用 AbstractRoutingDataSource 做分发，根据当前上下文决定用哪个数据源。

一般做法是定义一个@DataSource 注解，标记在 service 方法上，比如 @DataSource("slave1")。然后通过 AOP 拦截这些方法，把数据源标识存到 ThreadLocal 里。动态数据源在查询时从 ThreadLocal 拿标识，返回对应的数据源实例。

配置上，先定义两个 DataSource Bean，比如 masterDataSource 和 slaveDataSource，注册到一个 DynamicDataSource 里，再把这个 DynamicDataSource 放到 Spring 容器作为主数据源。事务管理器也得指定用这个动态数据源。

注意事务要小心，如果用了 @Transactional，默认绑定的主数据源，跨数据源事务搞不定，只能保证单个库的 ACID。

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface DataSource {
    String value();
}

// AOP 切面设置数据源
@Before("@annotation(ds)")
public void setDataSource(DataSource ds) {
    DynamicDataSourceContextHolder.set(ds.value());
}

# 说说你对SpringBoot事件机制的了解？

Spring Boot 的事件机制本质上就是发布-订阅模式，基于 Spring 框架的事件体系实现，核心作用是让应用内部的各个组件之间能够解耦通信。

整个事件流转过程是这样的：

  1. 发布者通过 ApplicationEventPublisher 把事件丢出去
  2. ApplicationEventMulticaster 负责把事件广播给所有注册的监听器
  3. 监听器收到事件后执行对应的处理逻辑。

所以，发布者压根不需要知道谁在监听，监听器也不用关心事件是谁发的，两边完全解耦。

实际开发中常见的使用场景有这么几个：

  1. 应用启动时做一些初始化操作，比如预热缓存、建立连接池、监听 ApplicationReadyEvent 就行
  2. 业务解耦，比如用户注册成功后要发送欢迎邮件、送新人券，把这些操作做成事件监听器，注册服务只管发事件，不用硬编码调用各种服务
  3. 异步处理耗时任务，配合 @Async 注解，事件处理可以放到单独的线程池里跑，不阻塞主流程

# SpringBoot2.x与1.x版本有哪些主要的改进和区别？

Spring Boot 2.x 相比 1.x 最核心的变化是底层升级到了 Spring Framework 5 带来了响应式编程的全面支持。除此之外还有几个关键改进：

  1. 底层框架升级 2.x 基于 Spring 5，支持响应式编程；1.x 基于 Spring 4，只能用传统的 Servlet 模型
  2. 嵌入式容器版本提升：Tomcat 从 8.x 升到 9.x，Jetty 维持在 9.x 系列但升级到了更高的小版本，Undertow 从 1.x 升到 2.x，性能更好，还支持 HTTP/2
  3. 默认连接池换成 HikariCP 1.x 默认用的是 Tomcat JDBC 连接池，2.x 换成了更轻量高效的 HikariCP，数据库连接性能提升明显
  4. Actuator安全策略调整：1.x 默认把所有监控端点都打开，有安全隐患；2.x 默认关闭大部分端点，需要显式配置才能暴露

# 什么是SpringActuator？它有什么优势？

Spring Boot Actuator 是 Spring Boot 自带的生产级监控工具，提供了一堆 HTTP 端点，用来查看应用的健康状况、性能指标、环境配置这些运行时信息。不用自己写代码，加个依赖配置一下就能用。

Actuator 暴露的端点分几类：

  • /health 用来做健康检查，Kubernetes 的探针就靠它。
  • /metrics 提供 JVM 内存、线程数、HTTP 请求耗时这些指标。
  • /env 能看到所有的配置项。
  • /loggers可以运行时动态调整日志级别。

它的优势：

1）开箱即用，加一行依赖就能跑起来，不用自己写监控接口 2）跟 Prometheus、Grafana 这些监控系统集成很方便，业界标准的 Micrometer 指标格式 3）端点可以自由开关，敏感信息不想暴露就关掉 4）支持自定义端点，业务特有的监控需求自己加

# 什么是SpringBoot？

Spring Boot，说白了就是 Spring 的脚手架或者是 Spring 的懒人包。

以前我们用 Spring 开发（SSM 时代），就像是自己买零件组装电脑：你需要自己去找各种 jar 包，还要担心版本冲突，然后写一大堆 XML 配置文件，最后还得自己装个 Tomcat 才能跑，非常繁琐，也就是大家常说的配置地狱。

而 Spring Boot 就像是直接买一台品牌整机（比如 MacBook）：起步依赖（Starters）它把常用的零件都打包好了（比如 web 开发套餐），你引入一个依赖，其他的它自动帮你配齐，而且版本都帮你管理好了，不会冲突。

自动配置（Auto-Configuration）这是它的核心。它遵循约定大于配置，比如你引入了 MySQL 的包，它就猜到你要连数据库，自动帮你把连接池配好，除非你非要自己改。

内嵌服务器 它自带了 Tomcat，打成一个 Jar 包，一行 java -jar 命令就能跑，不用再单独部署 Web 服务器了。

所以，Spring Boot 的本质就是：Spring + 自动化配置，目的是让我们只关注业务代码，而不是把时间浪费在配置上。

# 在 Spring Boot 中你是怎么使用拦截器的？

拦截器在项目里主要用来做权限校验、日志记录、性能监控这些横切关注点的事情。实现起来就两步：

1）实现 HandlerInterceptor 接口，重写里面的三个方法，分别对应请求前、请求后、整个请求结束后这三个时间点

2）写一个配置类实现 WebMvcConfigurer，在 addInterceptors 方法里把拦截器注册进去

先看拦截器的执行流程：请求进来先过 preHandle，返回 true 才会继续往下走到 Controller，Controller 执行完再走 postHandle，最后不管成功失败都会走 afterCompletion 做收尾。

# 如何理解SpringBoot中的starter？

Starter 本质上是一个依赖打包机制，把某个功能需要的所有依赖和默认配置捆在一起，引入一个 starter 就能开箱即用。

比如要用 Redis，以前得自己写 jredis 或 lettuce 客户端、spring-data-redis、连接池等一堆依赖，版本还得对得上。现在直接引 spring-boot-starter-data-redis，这些全给你配好了，application.yml 里写个 host 和 port 就能用。

# SpringBoot默认同时可以处理的最大连接数是多少？

拿 Spring Boot 默认内嵌的 Tomcat 来说，最大连接数 max-connections 默认 8192，再加上 accept-count 默认 100。

所以 SpringBoot 默认能同时接受的最大请求数是 8292。accept-count 是当所有处理线程都忙的时候，操作系统层面还能排队等待的连接数。

# SpringBoot打成的jar和普通的jar有什么区别？

Spring Boot 打出来的 jar 和普通 jar 最大的不同在于可执行性和依赖打包方式。

普通 jar 只是把 .class 文件按包结构打进去，运行得靠外部 classpath 引入依赖。你用 java -jar 跑一个普通 Spring 项目 jar，基本会报错找不到主类或者依赖。

Spring Boot 的 jar 是个胖 jar (fat jar)，它把项目所有的编译类 + 依赖的第三方库 + 内嵌容器（比如 Tomcat）全塞进一个 jar 里了。启动的时候，通过自定义的类加载器 LaunchedURLClassLoader 动态加载内部的 jar 包。

它的目录结构也特别，除了正常的 /classes，还有 /lib 目录存着所有依赖 jar，/BOOT-INF/classes 放项目自己的 class，/META-INF/MANIFEST.MF 里的 Main-Class 指向的是 org.springframework.boot.loader.JarLauncher，不是你写的 Application 类。

这个 JarLauncher 就是启动入口，它会先解析内部结构，把 /BOOT-INF/lib 下的 jar 全加到类路径，再反射调用你配置的真正启动类。

代码上不用你操心，只要在 pom.xml 用了 spring-boot-maven-plugin，build 时就会自动处理。

<build>
    <plugins>
        <plugin>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-maven-plugin</artifactId>
        </plugin>
    </plugins>
</build>

# 如何在SpringBoot中定义和读取自定义配置？

Spring Boot 读自定义配置，本质是把配置文件里的值映射到 Java 对象里。最常用的方式就是 @ConfigurationProperties，它能把配置自动绑定到类的字段上。

比如你在 application.yml 里加了：

app:
  name: user-service
  timeout: 5000

对应写个类：

@Component
@ConfigurationProperties(prefix = "app")
public class AppProperties {
    private String name;
    private long timeout;

    // getter/setter
}

只要加上 @Component，这个类就能在其他地方 @Autowired 注入使用。字段名会自动和配置项匹配，支持松散绑定（比如 timeoutMs 可以对应 timeout-ms）。

如果不想用 @Component，也可以在配置类里用 @Bean 手动注册，然后照样能用 @ConfigurationProperties 绑定。

还有一种方式是 @Value("${xxx}")，适合读单个值：

@Value("${app.timeout}")
private long timeout;
1
2

但 @Value 不支持复杂对象、不支持松散绑定、也不支持 JSR-303 校验，灵活性差很多。真要搞配置对象，还是得靠 @ConfigurationProperties。

它支持校验，比如加上 @Validated 就能用 @NotNull 这些注解。也支持嵌套结构，List/Map 类型也能处理。

# SpringBoot支持哪些嵌入Web容器？

Spring Boot 开箱就⽀持好⼏种嵌⼊式 Web 容器，最常⽤的其实是三个：Tomcat、Jetty 和 Undertow。

  1. 默认是 Tomcat，你只要引⼊ spring-boot-starter-web ，它⾃带了 Tomcat 作为嵌⼊容器，启动后就能直接对外服务，不⽤额外部署 WAR 包。
  2. 如果你想⽤ Jetty，⽐如在⾼并发⻓连接场景下压测表现更好，那就把 starter ⾥的 Tomcat 依赖排除，加上spring-boot-starter-jetty 。像 Spring Cloud Gateway 有些项⽬就会切到 Jetty 或 Undertow。
  3. Undertow 是 Red Hat 出的，性能不错，内存占⽤也低，适合想压榨服务器资源的场景。换起来也简单，排除Tomcat 后引⼊ spring-boot-starter-undertow 就⾏。它们都是通过 ServletWebServerFactory 接⼝来创建具体容器实例的，Spring Boot ⾃动配置会根据 classpath上的类选择合适的实现。⽐如看到 org.apache.catalina.startup.Tomcat 就上 Tomcat，看到io.undertow.Undertow 就初始化 Undertow。

代码上完全不⽤改，切换靠依赖管理。举个 Maven 的⽚段：

<exclusions>
    <exclusion>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-tomcat</artifactId>
    </exclusion>
</exclusions>

# SpringBoot是如何通过main方法启动web项目的？

Spring Boot 启动 Web 项⽬不是靠 main ⽅法本⾝多⾼级，⽽是靠 SpringApplication.run() 这个静态⽅法把⼀堆脏活累活给⼲了。main ⽅法⾥通常就两⾏代码，核⼼是调⽤ SpringApplication.run(Application.class, args) 。它会做⼏件事：推断应⽤类型，⽐如有没引⼊ Tomcat、Jetty 这些依赖；加载所有 META-INF/spring.factories ⾥配置的初始化器和监听器；最后创建对应的上下⽂容器。

如果是 Web 应⽤，它会创建 AnnotationConfigServletWebServerApplicationContext，这个上下⽂知道怎么启动内嵌 Web 服务器。在刷新阶段，会触发 ServletWebServerFactoryAutoConfiguration ⾃动装配，根据classpath 选⼀个可⽤的 Web 服务器，⽐如默认是 Tomcat。

  1. 先找有没有 ServletWebServerFactory 类型的 Bean
  2. 通过⼯⼚获取 WebServer 实例，⽐如 TomcatServletWebServerFactory
  3. 调⽤ webServer.start() 把内嵌容器跑起来
@SpringBootApplication

# Spring Boot 的核心特性有哪些？

Spring Boot 的核⼼特性其实围绕⼀个⽬标：让 Spring 应⽤能快速落地，不⽤在配置上折腾半天。

  1. ⾃动装配是它最硬的底⽓。你加个 spring-boot-starter-web ，它就帮你把 Tomcat、Spring MVC 都配好了，不需要写⼀堆 XML 或 @Configuration 。背后的逻辑是 @ConditionalOnClass 这些条件注解，只要类路径有某个类，就⾃动⽣效对应配置。
  2. 起步依赖（Starter）设计得很聪明。⽐如⽤ Kafka 就引⼊ spring-kafka-starter ，它把常⽤的依赖版本都锁定了，不会出现版本冲突。常⻅的像 spring-boot-starter-data-jpa 、 spring-boot-starteractuator 都是这种模式。
  3. 内嵌容器让部署变简单。以前要打 WAR 包扔到外部 Tomcat，现在直接 java -jar 启动，因为内嵌了Tomcat、Jetty 或 Undertow。请求进来时，Servlet 容器直接加载应⽤，压根不经过外部部署。
  4. Actuator 提供⽣产级监控。⽐如 /actuator/health 看健康状态， /actuator/metrics 查 JVM 指标，结合 Prometheus 和 Grafana 能搞监控⼤盘。
  5. 外部化配置⽀持多环境。 application.yml ⾥写 spring.profiles.active=prod ，就能切换不同配置，本地和线上互不影响。

代码⽰例就是最简单的启动类：

        	SpringApplication.run(App.class, args);

# Spring Boot 3.x 与 2.x 版本有哪些主要的改进和区别？

Spring Boot 3 是⼀次真正意义上的⼤版本跃进，不是简单功能叠加。最关键是它全⾯拥抱了 Java 17+ 和 Jakarta EE 9+，这意味着包路径从 javax.* 全⾯迁移到了 jakarta.* 。你如果⽤⽼的 javax.servlet 相关依赖，直接启动就报错，必须升级配套中间件。Java 版本要求从 8 跳到 17，官⽅推荐使⽤ LTS 版本开发。这带来了很多语⾔层⾯的新特性可⽤，⽐如 record、sealed class、模式匹配等，框架内部也⼤量使⽤这些特性简化代码。

GraalVM 原⽣镜像⽀持是另⼀个重头戏。Spring Boot 3 配合 Spring Native 可以把应⽤编译成原⽣可执⾏⽂件，启动时间从秒级降到毫秒级，内存占⽤减少⼀半以上。像云函数、Serverless 这种场景特别适合。

  1. ⾃动配置机制更智能，条件注解⽀持延迟评估，避免早期加载不必要的 Bean
  2. Micrometer 指标监控成为⼀等公⺠，默认暴露 /metrics 端点，集成 Prometheus 更顺畅
  3. 弃⽤了部分陈旧 API，⽐如 @RefreshScope 在 Kubernetes 场景下⽤ ConfigMap + Reloader 更合适

代码上变化不⼤，但⽣态组件要跟上。⽐如你⽤的是 Tomcat 9，得升到 10 才⽀持 Jakarta Servlet；⽤ JPA 的话 Hibernate 要 6+。

	return route(GET("/users/{id}"), handler::getById);
}
