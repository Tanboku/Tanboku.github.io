# Redis面试题

# Redis 中常见的数据类型有哪些？

字符串（String）是最基础的类型，能够存文本、序列化对象，也常用来做计数器。比如用 INCR 实现接口调用次数统计，简单高效。可以参考利用Redis实现高并发计数器_java redis高并发计数-CSDN博客 (opens new window) (opens new window)。

列表（List）有序,底层是双向链表，支持从头到尾的插入删除，适合做消息队列。但是随机访问慢，一般不推荐当作数组使用。LPOP/RPOP 配合 LPUSH/RPUSH 能实现简单的生产消费模型。

集合（Set）无序、不重复的字符串的集合，底层用哈希表来实现，所以增删查都是 O(1)。可以用在用户标签管理、社交关系去重这种需要快速判断成员存在的场景。

有序集合（Sorted Set）在 Set 基础上加了 score 排序，比如排行榜类需求，ZADD加数据，ZRANGE按排名取，性能很好。score 可重复，member 才是唯一键。

哈希（Hash）适合存储对象字段，比如用户信息，field对应属性名，value存值。相比于直接序列化整个对象，可以单独更新某个field，省宽带也省内存。

上面五种是日常最常用的，像是Stream、Geo、Bitmap 这些属于特定场景扩展类型，用的相对少一些。

# Redis 为什么这么快？

纯内存操作 + 单线程 + IO多路复用 + 高效的数据结构

首先，光内存操作就比磁盘快，举个栗子，访问一次内存大概120纳秒，访问一次SSD要50-150微秒，而传统的机械硬盘就更慢了。内存快了将近1000倍。Redis把数据全放到内存里，除了持久化，基本上不和磁盘打交道。但这样还不够，比如用Java写个Map存数据，也快，但扛不住并发。

所以让Redis同时处理几万连接的核心是单线程+IO多路复用模型。

单线程执行命令，没有锁竞争，没有上下文切换。不过前提就是命令执行必须足够快，所以KEYS *要少用，官方推荐scan。

网络IO用 epoll多路复用，详细可参考Linux epoll完全图解，彻底搞懂epoll机制 - 知乎 (opens new window) (opens new window)，一个线程就能同时同时盯着几万个socket连接，哪个连接有数据就处理哪个，不用等。

在数据结构上，Redis直接基于C语言实现紧凑结构，比如SDS字符串比普通字符串少内存拷贝，哈希表查找接近O(1)。

# 为什么 Redis 设计为单线程？6.0 版本为何引入多线程？

Redis是对内存操作，走的是CPU总线，但是Redis的瓶颈不在CPU，而在于内存和网络IO，一道GET/SET指令，CPU执行时间是微妙，网络返还才是大头。

用单线程的好处很多，没有锁竞争，简单，没有上下文切换，搭配IO多路复用官方测试能跑到10万+QPS，大多数的业务场景够用。

6.0版本引入多线程，主要是给网络IO擦屁股，在并发量高的情况下，单线程处理不过来，CPU还不能被充分利用。要多线程承担网络读写的工作，不过主线程命令执行还是单线程，所以不用担心线程安全问题，还能提升吞吐量。

# Redis 中跳表的实现原理是什么？

跳表本质上是一个多层链表，底层链表保存着所有的元素，上层链表是下层链表的子集，查找时先从上层开始往右找，遇到比目标值大的就往下走一层，比如要找50，最上层链表是40 -> 70，这样就需要往下走一层，拿空间换时间，时间复杂度为O(logn)。插入就是正常插入，然后在随机决定建立基层索引，Redis是25%。删除就是多层多删几次。

# Redis 的 hash 是什么？

是一种键值对的集合，可以把多个字段和值存储到同一个key下面，特别适合存储对象属性，比如用 user:0001 作为key，里面放age，name这些字段，比 json 灵活不少。

Hash底层有两种编码方式，数据量小的时候用 listpack 存储，紧凑省内存；数据量大了自动切换成 hashtable ，查询效率O(1)。这种转换是单向的，切换阈值默认是512个字段或者单个值超过64字节。

Redis的哈希表扩容不是一次性搬完的，而是分多次慢慢搬，这就是渐进式 rehash。先弄一个新表，大小为2倍，更改rehash字段值-1到0，每次增删改查慢慢搬。

# Redis Zset 的实现原理是什么？

Zset底层是跳表 + 哈希表的双结构配合，跳表按照score排序，哈希表存储member到score的映射，通过member查score走哈希，按score范围查member走跳表，插入删除同时更新两个数据结构。

元素少的时候用 listpack 存储，当元素个数超过128且每个元素长度超过64字节，自动升级成哈希表 + 跳表的形式，阈值可以配置。

# Redis 中如何保证缓存与数据库的数据一致性？

缓存和数据库的双写一致性的问题本质上是分布式系统下的两个数据源的同步问题。常用的有三种：

先更新数据库，再删缓存，在极端的情况下还是会出现数据不一致的问题，不过概率很小。

先删缓存，更新数据库，再删缓存，也就是延迟双删。

用Canal监听MySQL Binlog，通过消息队列异步更新缓存。

# Redis 中的缓存击穿、缓存穿透和缓存雪崩是什么？

这三个问题都是缓存失效导致请求打到数据库的场景，区别在于失效的原因和范围不同。

缓存穿透：查不存在的数据，缓存里没有就会去查数据库，并发状态下对db压力很大。可以缓存空数据或者上布隆过滤器。

缓存击穿：某个热点key过期，这时候大量的并发请求同时打了过来，对db压力很大。可以设置逻辑过期，过期后可以先返回旧数据再慢慢重建缓存。

缓存雪崩：大量的key同时过期，或者Redis挂了。可以设置随机过期时间。

# Redis String 类型的底层实现是什么？（SDS）

Redis 里的 String 类型底层其实用的不是 C 语言原生的 char 数组，而是自己搞了一套叫 SDS（Simple Dynamic String）的数据结构。这么做主要是为了解决 C 字符串在实际使用中的痛点，比如获取长度得遍历、没法安全拼接、容易缓冲区溢出这些。

SDS 的结构很简单，核心就三个字段：len 记当前用了多少字节，free 记后面还剩多少空闲空间，buf[] 是真正的字符数组。因为存了 len，所以 strlen 这种操作是 O(1) 的，压根不经过遍历。拼接字符串前它会先看 free 够不够，不够就扩容，避免了溢出问题。

还有个关键点是 SDS 在修改时会做空间预分配和惰性释放。比如你追加内容，它不会刚好分配你需要的大小，而是多分一点，下次再改就能省掉一次内存调整。删数据时也不立刻 free，留着 free 字段里，方便后续复用。

# Redis 中如何实现分布式锁？

在多线程的场景下，可能会导致一些问题，这时候就可以加锁，让线程一个个处理，这就是锁，分布式锁就是分布式场景下的锁。

实现分布式锁，最朴素的想法是用一个能共享的状态来标记“锁被谁占了”，Redis 天然适合这个场景。关键不是 set 本身，而是怎么保证 set 的原子性和后续的可靠性。

加锁时，必须用一条原子命令完成设置值和过期时间，不能分开写（避免 set 成功但 expire 失败导致死锁）。标准写法是：

SET resource_name random_value NX PX 30000

基础方案存在锁过期时间不好定的问题——设短了业务没执行完锁就自动释放，设长了万一宕机锁又卡死。生产环境常用 Redisson 客户端，它封装了可重入锁和看门狗机制。当调用 lock() 时，如果未指定超时时间，Redisson 会默认设置 30 秒过期，并启动一个后台线程每隔 10 秒检查一次：只要业务线程还活着，就自动将锁的过期时间续到 30 秒，确保锁不会在业务执行中提前释放。若指定了 tryLock(10, TimeUnit.SECONDS)，看门狗则不会介入，过期时间由开发者自行控制。

上述方案在单实例下没问题，但若 Redis 采用主从架构，主节点刚写入锁还未同步到从节点就宕机了，从节点升主后，新主没有这把锁，其他客户端就能重新加锁，导致同一资源被同时持有（锁失效）。Redlock 算法针对此问题提出：客户端在多个（通常 5 个）互相独立的 Redis 节点上依次申请锁，只有超过半数节点加锁成功且总耗时小于锁过期时间，才认为获取成功。不过该算法在实际生产中存在争议（依赖时钟同步、网络延迟等），一般中小公司用单实例 + 高可用部署配合看门狗已能满足大部分场景，高安全场景则建议转向 ZooKeeper 或 etcd。

# Redis 的 Red Lock 是什么？你了解吗？

Red Lock 是 Redis 官方提出的一种分布式锁实现方案，目标是解决单实例 Redis 主从切换时锁失效的问题。它不依赖单一 Redis 节点，而是要求客户端在多个独立的 Redis 实例上申请锁，只有大多数节点都加锁成功才算真正获取到锁。

但 Red Lock 争议很大。Martin Kleppmann 等人指出，它严重依赖系统时钟的稳定性。如果某个客户端因为 GC 停顿导致时间跳跃，就可能误判锁的有效性，从而引发多个客户端同时持有同一把锁。相比之下，像 ZooKeeper 或 etcd 这类基于共识算法（如 ZAB、Raft）的协调服务，在处理分布式锁时更安全。它们通过 leader 选举和顺序写入来保证强一致性，不需要依赖本地时钟。一般中小公司若对性能要求极高，可用 Redis 单实例 + 高可用部署 + 看门狗机制满足大部分场景；若业务对强一致性有极致要求（如金融转账），则建议直接上 ZooKeeper 或 etcd。

# Redis 实现分布式锁时可能遇到的问题有哪些？

1）锁过期时间不好定 业务执行时间是不确定的。如果过期时间设短了，锁自动释放了但业务还没执行完，其他客户端就能抢到锁，导致数据错乱；如果设长了，万一持有锁的节点宕机，锁迟迟无法释放，整个系统就可能卡死。

2）不是原子操作就危险 加锁时如果分开执行 SET key 和 EXPIRE，中间一旦 Redis 宕机，EXPIRE 没执行，锁就成了“永久锁”，永远不会释放。一定要用 SET key value NX PX milliseconds 这一条原子命令搞定。

3）主从切换导致锁失效（脑裂） Redis 主节点刚写入了锁，还没来得及同步给从节点就挂了。随后从节点被提升为新主，这时新的客户端请求在新主上又能成功加上同名锁。这就导致同一个资源被两个客户端同时持有，失去了互斥性，也就是锁的“脑裂”问题。

4）误删别人的锁 业务执行超时，当前客户端持有的锁已经过期并被其他客户端获取。如果此时当前客户端直接执行 DEL key，就会把别人刚刚获得的锁删掉。正确做法是：设置 value 为全局唯一标识（如 UUID + 线程 ID），删除前先通过 Lua 脚本 GET 判断是否为自己持有，匹配才删除，确保“判断 + 删除”是原子操作。

5）Redlock 算法争议大 虽然 Redlock 旨在解决单点故障问题，但它依赖系统时钟，且实际场景中的网络延迟、GC（垃圾回收）停顿都可能让它的安全性打折扣，Martin Kleppmann 等专家对此提出过质疑。一般中小公司使用 Redis Cluster 配合带唯一值的 SET 和 Lua 脚本删除就足够了，追求高可靠性的场景建议直接上 ZooKeeper 或 etcd。

# Redis 的持久化机制有哪些？

Redis 提供了两种主要的持久化机制：RDB（快照）和 AOF（追加日志文件），生产环境通常建议两者同时开启，以兼顾备份恢复速度和数据安全性。

1）RDB（Redis DataBase）—— 全量数据快照

RDB 是某个时间点的数据全量快照。它会 fork 一个子进程，把当前内存数据全量写入 RDB 文件。文件紧凑，恢复速度快，适合做备份。比如用 Redis 做缓存集群时，每天凌晨生成一次 RDB，相当于一次“全量归档”。但两次快照之间如果宕机，中间的数据就丢了，所以无法做到完全不丢数据。

2）AOF（Append Only File）—— 写操作命令日志

AOF 记录的是每一条写操作命令，类似数据库的 binlog，以追加方式写入 aof 文件。通过配置 fsync 策略控制刷盘频率。appendfsync everysec 最常用，性能和安全性比较均衡，最多丢 1 秒数据。不过 AOF 文件通常比 RDB 大，恢复也慢一些，因为要重放所有命令。AOF 重写机制可以定期压缩日志，去掉无效操作（如多条 INCR 合并为一条 SET），避免文件无限膨胀。

生产环境一般同时开启 RDB 和 AOF。Redis 重启时优先使用 AOF 恢复数据，因为 AOF 数据更完整；若 AOF 文件损坏或不存在，则加载 RDB 文件。这种组合方式能在恢复速度和数据完整性之间取得较好的平衡。

# Redis 主从复制的实现原理是什么？

Redis 主从复制是靠主节点把写操作同步给从节点来实现数据一致的。整个过程分全量同步和增量同步两个阶段，大部分情况下两者会配合使用。

1）全量同步（初次建立复制或断连严重时触发）

第一次建立复制关系或从节点落后太多时，走全量同步。从节点发起 psync 命令，带上自己的 replicationID 和复制偏移量 offset，主节点发现不匹配就会触发全量同步。主节点会 fork 一个子进程生成 RDB 快照，同时用**复制缓冲区（replication buffer）**记录这期间的所有新写命令。RDB 文件传输给从节点后，从节点先加载恢复数据，然后主节点再把缓冲区内积压的写命令发给从节点，确保数据完整追平。

2）增量同步（常规运行或短暂断线重连）

全量同步完成后进入增量同步阶段。主节点每执行一个写命令，都会通过网络把命令实时传播给所有从节点。从节点按相同顺序执行这些命令，保持和主节点状态一致。如果网络短暂中断，从节点重连后，可以通过主节点上的**复制积压缓冲区（repl_backlog_buffer）**中的积压日志快速追平断连期间的增量数据，而不需要重新触发全量同步，大大降低了恢复成本。

3）心跳检测与容错

从节点每隔 1 秒发送 REPLCONF ACK 给主节点，报告自己的复制偏移量。主节点借此感知从节点是否存活，并监控复制进度。这种心跳机制让复制具备一定容错能力，为 Redis Sentinel 哨兵或 Cluster 集群模式下的故障转移提供了判断依据。

关键注意事项： 复制是异步的，从节点可能存在秒级延迟，无法保证实时强一致。此外，如果主节点持久化关闭（不开启 RDB/AOF），一旦主节点宕机重启，从节点会同步到一份空数据，造成雪崩式数据丢失。生产环境务必开启主节点 AOF，并设置 min-slaves-to-write（最小从节点写入数）和 min-slaves-max-lag（最大延迟阈值）来防止脑裂及数据丢失。

# Redis 数据过期后的删除策略是什么？

Redis 的过期删除策略采用惰性删除和定期删除两种机制相结合的方式，两者互补，共同维持内存健康。

  1. 惰性删除：在访问 Key 的时候才检查它是否已过期，若过期则立即删除并返回 nil。比如执行 GET 操作时，Redis 会先检查该 Key 是否存在且已过期，如果过期就删除掉。这种方式虽然简单直接，但存在一个问题：如果一个 Key 过期了却一直没人访问，它就会一直占用内存，成为“僵尸 Key”。
  2. 定期删除：为了解决惰性删除可能遗留过期 Key 的问题，Redis 会周期性地主动清理。它会随机抽查一部分设置了过期时间（TTL）的 Key，删除其中已经过期的。默认情况下，Redis 每秒运行 10 次清理任务，每次最多处理 25 个 Key（具体数量可配置），并且只会扫描设置了过期时间的字典，以此在不大幅影响性能的前提下，主动清理掉一批过期 Key。

这两种机制是互补关系：定期删除负责兜底，将大部分过期 Key 批量清理掉；惰性删除则作为最后一道防线，确保任何一个过期 Key 在被访问时都不会被错误地返回给客户端。

在实际场景中，像使用 Redis 做缓存或会话存储（如 SpringSession + Redis）时都依赖这套机制。如果你发现内存增长异常，很可能就是因为过期 Key 没有被及时删除，此时需要检查是否因为写入远多于读取，导致惰性删除触发频率过低。

# 如何解决 Redis 中的热点 key 问题？

热点 key 问题说白了就是某个 key 被疯狂访问，QPS 可能上十万，直接把 Redis 实例的 CPU 或网卡打满，甚至拖垮整个服务。

1）先定位。一般通过 redis-cli --hotkeys 或监控系统（比如用 Prometheus + Grafana）看 key 的访问频率。生产环境建议在 proxy 层（比如 TendisProxy 或 Codis）埋点统计 top N 热 key。

2）解决思路不是单一的，得组合拳。本地缓存是第一道防线。应用节点用 Caffeine 或 Guava Cache 把热 key 缓一份，设置短过期时间（比如 1~2 秒），这样大部分请求就压根不经过网络到 Redis。注意缓存穿透问题，可以用布隆过滤器兜底。

3）拆分 key 也是常用手段。比如原来一个 key 存全量数据，改成 key_1、key_2 分片存储，访问时随机或轮询拉取，把压力分散。适用于静态资源类场景，比如首页 banner。

4）Redis 侧可以做多副本。比如给这个 key 单独起一个只读副本集群，或者利用 Redis Cluster 的 replica 只读节点分流读请求。

5）极端情况，写扩散。比如把热 key 的值推送到所有客户端内存里，靠消息中间件（比如 Kafka）通知更新。这招重运维，搞不定一致性别轻易上。

# Redis 集群的实现原理是什么？

Redis 集群的核心是数据分片和去中心化协作。它把整个数据空间切成 16384 个槽（Slot），每个键通过 CRC16 算出哈希值后对 16384 取模，落到对应的槽上。集群里的每个主节点负责一部分槽，比如节点 A 管 0~5000，节点 B 管 5001~10000，节点 C 管 10001~16383。

客户端可以直接连任意节点查数据，如果请求的 key 不归这个节点管，它会返回一个 MOVED 指令，告诉客户端该去哪个节点找。聪明的客户端会缓存 slot 和节点的映射关系，下次直接访问目标节点，压根不经过转发。

节点之间通过 Gossip 协议互相通信，每秒交换几次信息，同步自己负责的槽、健康状态、新节点加入等。这种设计没有中心节点，靠多数主节点投票完成故障转移。比如某个主挂了，它的从节点发现后会发起选举，拿到半数以上同意就能顶上。

集群也支持动态扩缩容。加新节点时先分配槽，再把旧节点上的槽迁过去，迁移过程不影响服务。可以用 redis-cli --cluster 工具自动完成。

需要注意的是，集群不保证强一致性，异步复制可能导致少量数据丢失。另外多 key 操作必须都在同一个 slot，否则会报错。要用的话得靠 hash tag 强制让多个 key 落到同一槽，比如 {user1000}.name 和 {user1000}.age。

# Redis 中的 Big Key 问题是什么？如何解决？

Big Key 指的是 Redis 中单个 key 对应的 value 过大，比如一个 String 超过 10MB，或者一个 Hash、ZSet 成员数量超过几万个。这种 key 会带来一系列连锁问题：

  1. 网络传输瓶颈：一次读取或写入 Big Key 可能占满带宽，导致其他请求被阻塞。
  2. 主从复制缓慢：在 RDB 或 AOF 重放阶段，Big Key 的加载会拖慢整个恢复过程，延长故障切换时间。
  3. 阻塞主线程：Redis 是单线程处理命令，Big Key 的删除操作（尤其是 DEL）可能引发长时间阻塞，甚至触发延迟告警。如果内存分配或释放时涉及大块连续空间，也可能导致瞬间卡顿。

常见的解决方案有以下几个方向：

  • 拆分数据结构：把一个大结构拆成多个小 key。例如，原来用一个 Hash 存百万用户积分，现在按用户 ID 分片，使用 user:score:001、user:score:002 这样分散存储，读写时通过计算定位到具体 key，避免单一大结构操作。
  • 压缩数据：如果 value 是文本或 JSON，可以考虑用 gzip 等算法压缩后再存入，取出时再解压。虽然会增加 CPU 开销，但能显著降低内存占用和网络传输量。
  • 选用合适的数据类型：比如原本用 String 存序列化的大对象，不如拆成多个 field 用 Hash 存储，这样可以支持部分字段的更新，避免每次全量读写。
  • 控制批量操作粒度：避免一次性 HGETALL 一个超大 Hash，可以分批获取部分字段，例如每次只取少量 field，减少单次命令的负载。

在监控层面，可以使用 redis-cli --bigkeys 定期扫描，结合 Redis 自身的慢日志（slow log）和内存分析工具及时发现隐患。代码层面也要注意控制批量操作的粒度，比如分批读取 Hash 字段，而不是一口气全量拉取。
