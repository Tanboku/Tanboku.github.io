# 设计模式面试题

# 单例模式有那几种实现？如何保证线程安全？

单例模式常见有：

  • 饿汉式：实例在类加载时就创建，线程安全，但如果实例初始化较重或没有被使用会浪费资源。
  • 懒汉式（线程安全版需要加双重检查锁定）：实例在首次访问时创建，节约资源，但需要确保线程安全。
  • 双重检查锁定：在懒汉式的基础上优化，直接加锁效率太低，双重检查锁只在第一次检查实例为空时加锁，提高性能。
  • 静态内部类：利用类加载机制实现懒加载和线程安全，推荐使用。
  • 枚举单例（Java特有）：通过枚举实现单例，简单且防止反射和序列化攻击。

# 什么是策略模式？一般用在什么场景？

策略模式是一种行为型设计模式，核心思想是把一系列算法封装成独立的类，让它们可以互相替换。调用方只依赖抽象接口，压根不用关心具体用哪个算法。

代码里满屏的 if-else、switch-case 来选算法？用策略模式一拆，每个分支变成一个策略类，调用方想用哪个就注入哪个，干干净净。

策略模式有 3 个角色：

  • Strategy：策略接口，定义算法的抽象方法
  • ConcreteStrategy：具体策略，实现不同的算法逻辑
  • Context：上下文，持有策略接口引用，把实际工作委托给策略对象

# 什么是模板方法模式？一般用在什么场景？

模板方法模式是一种行为型设计模式，核心思想是在抽象类中定义算法骨架，把可变的步骤延迟到子类实现。父类定义了执行顺序和通用逻辑，子类只负责填充具体细节。

整个模式由两个角色组成：

AbstractClass：是抽象类，里面有个final修饰的模板方法定义算法骨架，调用一系列步骤方法。其中有些步骤是抽象方法强制子类实现，有些是默认实现子类可选覆盖。

ConcreteClass：是具体子类，实现抽象方法，按需覆盖钩子方法。

# 谈谈你了解的最常见的几种设计模式，说说他们的应用场景

工作中用得最多的就是单例模式、简单工厂、策略模式、模板方法这几个。

单例模式的场景很明确：全局只需要一个实例的时候。数据库连接池用的就是单例，HikariCP、Druid 这些连接池都是单例管理的。配置中心的客户端也是单例，不可能每次读配置都 new 一个新的。

策略模式和模板方法在业务开发中经常组合使用。拿支付场景举例：对接支付宝、微信、银联这些渠道，每个渠道的支付逻辑不一样，天然就是不同的策略。定义一个 PayService 接口，每个渠道实现自己的 pay 方法。

同时这些渠道的支付流程又有共性：都要先做参数校验，然后执行核心支付逻辑，最后做后置处理。把这个共性流程抽成模板方法，让各渠道实现类继承它，只重写核心支付那一步就行。

# 你认为好的代码应该是什么样的？

  1. 良好的扩展性
  2. 易读性
  3. 实现代码复用
  4. 良好的可维护性
  5. 简洁性
  6. 各种命名规范性
  7. 减少硬编码

# 工厂模式和抽象工厂模式有什么区别？

工厂模式和抽象工厂模式的核心区别在于创建对象的维度不同。

工厂模式关注的是创建单一类型的对象，定义一个抽象方法，由子类决定实例化哪个具体产品。一个工厂只生产一种产品。

抽象工厂模式关注的是创建一族相关对象，提供一个接口来创建一组相关或互相依赖的对象。一个工厂负责生产一整套配套的产品。

打个比方，工厂模式像是一条生产线只生产手机，换个型号就换条生产线。

抽象工厂模式像是一个品牌工厂，苹果工厂生产 iPhone + AirPods + MacBook 一整套苹果系产品，三星工厂生产 Galaxy + Galaxy Buds + Galaxy Book 一整套三星系产品。

产品之间有关联性，不能混搭。
